# Spec for template_system

## Summary

Набор шаблонных файлов в папке `test/`, позволяющий разработчику описать новый REST API
в 10-минутной анкете и получить готовую структуру Python-проекта: библиотеку-клиент,
CLAUDE.md, демо-ноутбук и шаблон спецификации функции.

Система рассчитана на API, по структуре похожие на Ozon Performance API:
аутентификация → получение списка сущностей → асинхронные или синхронные эндпоинты данных → вывод DataFrame.

Поддерживаемые варианты:
- **Аутентификация**: OAuth 2.0 `client_credentials` ИЛИ статический API-ключ (заголовок или query-параметр)
- **Паттерн эндпоинта**: асинхронный (submit UUID → poll → download) ИЛИ синхронный (прямой GET/POST)
- **Батчинг**: настраиваемый размер батча (или отключён)
- **Rate limiting**: экспоненциальный backoff при 429 (количество повторов и базовый интервал — настраиваемы)
- **Кумулятивные метрики**: паттерн отдельного запроса на каждый день (опционально)

| `template_CLAUDE.md` → | `CLAUDE.md` | корень проекта |

## Рабочий процесс генерации

1. Скопировать `test/NEW_PROJECT_QUESTIONNAIRE.md` → заполнить все поля.
2. Передать заполненную анкету Claude (или разработчику).
3. Claude подставляет значения всех `{PLACEHOLDER}` в:
   - `template_client.py` → создаёт `{MODULE_NAME}.py`
   - `template_CLAUDE.md` → создаёт `CLAUDE.md`
   - `template_demo.ipynb` → создаёт `{MODULE_NAME}_demo.ipynb`
4. Для каждой публичной функции Claude создаёт `specs/{NN}_spec_{SPEC_FUNCTION_NAME}.md`
   из `template_spec.md` и затем `plans/{NN}_plan_{SPEC_FUNCTION_NAME}.md` из `template_plan.md`.
5. Claude удаляет блоки вариантов, не подходящие под ответы анкеты
   (например, убирает блоки `[OAUTH ONLY]`, если API использует API-ключ).
6. Разработчик проверяет результат с реальными учётными данными.

## Правила подстановки плейсхолдеров

- Все плейсхолдеры используют синтаксис `{UPPER_SNAKE_CASE}`.
- Блоки вариантов аннотируются: `# ── [OAUTH ONLY]`, `# ── [ASYNC ONLY]`, `# ── [SYNC ONLY]`, `# ── [API KEY ONLY]` — Claude удаляет неподходящие блоки целиком.
- Плейсхолдер, оставшийся без подстановки, означает: «разработчик должен решить сам».
- Одно и то же имя плейсхолдера используется во всех шести файлах без исключения.

## Функциональные требования

### 1. Полнота анкеты
Анкета должна содержать достаточно информации для генерации `{MODULE_NAME}.py`
без дополнительных вопросов, при условии что API ведёт себя как стандартный REST API.

### 2. Корректность шаблона клиента
`template_client.py` после подстановки и удаления неиспользуемых блоков должен:
- быть синтаксически валидным Python (`python -m py_compile`)
- успешно импортироваться (`import {MODULE_NAME}`) при установленных зависимостях
- возвращать пустой `pd.DataFrame` с правильными колонками при отсутствии данных

### 3. Покрытие обоих вариантов
Каждый вариант (OAuth/API-ключ, async/sync) должен тестироваться независимо
из одного шаблона без остаточного мёртвого кода после удаления блоков.

### 4. Точность CLAUDE.md
`template_CLAUDE.md` должен включать имя каждой публичной функции, её параметры
и эндпоинт API — чтобы Claude Code мог ответить на вопрос «что это делает?»
только по CLAUDE.md, не читая исходный код.

### 5. Запускаемость демо-ноутбука
После подстановки учётных данных демо-ноутбук должен выполняться от начала до конца
(каждая ячейка с кодом запускается без ошибок импорта или синтаксиса).

## Acceptance Criteria

- [ ] Заполнение анкеты для произвольного API X (например, Google Ads API)
      даёт `{MODULE_NAME}.py`, проходящий `python -m py_compile`
- [ ] Сгенерированный `CLAUDE.md` содержит все константные значения из Block 3 `{MODULE_NAME}.py`
- [ ] Демо-ноутбук импортирует из `{MODULE_NAME}` без ошибок
- [ ] `template_spec.md` для новой функции содержит таблицу ограничений,
      согласованную с константами `{MODULE_NAME}.py`
- [ ] Все шесть шаблонных файлов внутренне согласованы: одни и те же имена
      плейсхолдеров разрешаются в одни и те же значения во всех файлах
- [ ] Вариант OAuth: нет ссылок на `{API_KEY_HEADER}`; вариант API-ключ: нет `_token`/`_refresh_token`
- [ ] Вариант async после удаления sync-блоков: нет ссылок на UUID/polling в sync-варианте;
      после удаления async-блоков: нет методов submit/poll/download

## Possible Edge Cases

- API использует OAuth для аутентификации И дополнительный API-ключ для некоторых эндпоинтов:
  в `template_client.py` добавить третий блок `# ── [HYBRID AUTH]`
- API не имеет списка сущностей (данные запрашиваются напрямую по диапазону дат без батчинга):
  C1 анкеты = «нет», блок `_fetch_all_{ENTITY_NAME}s()` помечен `# ── [BATCHING — удалить если не нужно]`
- API использует cursor-based пагинацию вместо диапазонов дат:
  D6 анкеты описывает пагинацию; `template_client.py` содержит заглушку `_paginate()`
  с пометкой `# ── [PAGINATION — добавить если нужно]`
- Числа в стандартном формате (точка как разделитель): F1 анкеты = «Нет»;
  `_parse_num()` упрощается до `float(value)`, замена запятой убирается
- Даты уже в формате ISO 8601: F2 анкеты = «Нет»; `_parse_date_str()` удаляется
