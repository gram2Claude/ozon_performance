# Порядок работы с шаблонами

## Шаг 0 — `01_PROJECT_STRUCTURE.md`
Инициализация нового проекта: создание папок и базовых файлов.
Скажи Claude: «Инициализируй структуру проекта».

Claude создаёт подпапку `{MODULE_NAME}/` в текущей директории и помещает туда только проектные файлы
(библиотека, CLAUDE.md, requirements, info/, specs/, plans/, smoke_tests/, raw_data/).
`manual_forms/` и `auto_generated/` остаются на уровне репозитория рядом с `test/` — они входные
данные и промежуточные артефакты, а не часть проекта.

---

## Шаг 1 — `manual_forms/02_NEW_PROJECT_QUESTIONNAIRE.md`
Заполни вручную только секцию A:
- **A1** — название API (человекочитаемое)
- **A2** — имя модуля (snake_case)
- **A3** — URL документации API

Всё остальное (базовый URL, тип авторизации, паттерн запросов, лимиты и т.д.)
Claude определяет автоматически из документации по ссылке A3.

---

## Шаг 1.5 — `info/00_api_methods.md` (генерируется автоматически)

После заполнения анкеты Claude **изучает документацию API** по ссылке из A3
(`{DOCS_URL}`) и создаёт в проекте папку `info/` с файлом `info/00_api_methods.md` —
сводку методов API.

Структура файла берётся из шаблона
`test/manual_forms/02b_API_METHODS_SUMMARY.md` (он же — образец оформления).

В сводку включаются **только**:
- Методы получения **статистики** (показы, клики, расходы, охваты, отчёты и т.п.)
- Методы **аутентификации** — выпуск и обновление access-токенов

**Не включаются** методы создания/редактирования рекламных кампаний,
управления объявлениями, биддинга, каталога товаров и т.п.

Для каждого метода Claude фиксирует:
- Оригинальные имена полей запроса/ответа (как в доках, без перевода)
- Типы данных (string, integer, number, array, object, ...)
- Формат ответа (`application/json`, `text/csv`, `text/tab-separated-values`,
  `application/zip` с CSV/TSV внутри, и т.п.)
- Кодировку и разделитель (для CSV/TSV)
- Sync/async-характер метода, rate limits, ограничения диапазона дат

Сводка используется на Шаге 2 (`03_ENTITY_FUNCTIONS.md`) для выбора набора
функций и на Шаге 3 при генерации скаффолда клиента.

## Шаг 3 — `auto_generated/` (генерируется автоматически)
На основе шагов 1 и 2 Claude создаёт **только скаффолд** (без концретных публичных функций):
- `04_template_client.py` → класс `{CLASS_NAME}` (auth, HTTP-обёртки),

## Шаг 4 — цикл реализации каждой функции

Функции реализуются **по одной**, в порядке из `03_ENTITY_FUNCTIONS.md`.
Рекомендуемый порядок: сначала справочники → статистика → охват.

### Перед первой функцией — однократный запрос настроек

Перед запуском smoke-теста **самой первой** функции Claude:
1. **Создаёт `.env`** (если не существует) — копирует из `.env.example`.
2. **Запрашивает у пользователя только значения** `CLIENT_ID`/`CLIENT_SECRET`
   (или API-ключа — в зависимости от типа авторизации). Файл `.env` к этому
   моменту уже создан; пользователю не нужно разбираться, куда вписывать.
3. **В том же запросе — запрашивает значения дат по умолчанию**, которые будут
   использоваться во всех последующих smoke-тестах:
   - `global_start_date` — если в проекте есть охватные/кумулятивные функции
   - `start_date` (он же `date_from`) — начало тестового периода
   - `end_date` (он же `date_to`) — конец тестового периода

   Claude **сам вписывает** полученные значения в `.env` —
   `TEST_GLOBAL_START_DATE`, `TEST_START_DATE`, `TEST_END_DATE` —
   и переиспользует их во всех последующих smoke-тестах автоматически.
   Повторно спрашивать запрещено.

### Источник информации об API при разработке — приоритет

При создании spec/plan и реализации функции Claude **сначала** обращается к
`info/00_api_methods.md` — ищет нужный метод (endpoint, поля ответа, типы,
формат, лимиты, авторизация).

- **Если метод найден** — использует данные оттуда. В интернет не ходит.
- **Если метода нет** — открывает официальную документацию API
  (URL берёт из поля **A3** файла `manual_forms/02_NEW_PROJECT_QUESTIONNAIRE.md`,
  не спрашивает пользователя), забирает нужное и **сразу обновляет
  `info/00_api_methods.md`** (добавляет блок о найденном методе по структуре
  шаблона `manual_forms/02b_API_METHODS_SUMMARY.md`). После обновления
  продолжает работу над функцией.

**Why:** экономия времени и токенов — не ходим повторно в интернет за тем,
что уже однажды было задокументировано. Файл `00_api_methods.md` остаётся
актуальным как единая точка правды по API проекта.

### Цикл для каждой функции

1. Claude создаёт спецификацию по шаблону `spec_templates/07_template_spec.md`
   → сохраняется в `specs/NN_spec_имя_функции.md`
2. Пользователь проверяет, вносит правки, утверждает
3. Claude создаёт план по шаблону `plan_templates/08_template_plan.md`
   → сохраняется в `plans/NN_plan_имя_функции.md`
4. Пользователь проверяет, вносит правки, утверждает
5. Claude реализует функцию согласно утверждённому плану
6. Проверка с реальным API:
   - Claude создаёт smoke-test скрипт `smoke_tests/test_<имя_функции>.py`
     в папке `smoke_tests/` проекта (если ещё не существует), читающий `.env`
     через `python-dotenv` из корня проекта (`PROJECT_ROOT / ".env"`)
   - Скрипт добавляет корень проекта в `sys.path` для импорта `{MODULE_NAME}`:
     ```python
     PROJECT_ROOT = Path(__file__).resolve().parent.parent
     sys.path.insert(0, str(PROJECT_ROOT))
     ```
   - **Запускает тест самостоятельно** через
     `python smoke_tests/test_<имя_функции>.py` (запуск из корня проекта) —
     не ждёт ручного запуска
   - Использует даты из `.env`: `TEST_START_DATE`, `TEST_END_DATE`,
     `TEST_GLOBAL_START_DATE` (для охватных функций)
   - Для функций с асинхронным API (submit→poll→download): передаёт
     `raw_cache_dir=PROJECT_ROOT / "raw_data" / "raw_files"` — сырые байты каждого отчёта
     сохраняются как `raw_data/raw_files/raw_{date_from}_{date_to}_{campaign_id}_{day}.csv`.
     При повторном запуске (например, после фикса парсера) данные берутся
     из кэша — API не вызывается.
     При запросе других дат старые `raw_*.csv` автоматически удаляются.
     Если несколько функций используют одни даты (например, `get_campaigns_daily_stat`
     и `get_ads_daily_stat`), они разделяют один кэш — вторая функция API не вызывает.
   - При ошибках чинит итеративно до зелёного результата
   - **После успешного прогона** сохраняет DataFrame в `raw_data/<имя_функции>.csv`,
     где `<имя_функции>` — точное имя Python-функции:
     - справочники: `get_campaign_dict.csv`
     - статистика с датами: `get_campaigns_daily_stat_2026-04-24_2026-04-25.csv`

   **Зачем `smoke_tests/` отдельно от `tests/`:** `tests/` — pytest unit-тесты
   с моками (быстрые, без `.env`). `smoke_tests/` — реальный API + `.env`
   (минуты, требуют credentials). Разделение исключает конфликт с pytest discovery.
7. **Обязательный показ результата:** после успешной проверки Claude **обязан
   вывести в чат первые 5 строк** из **сохранённого CSV-файла** (`raw_data/...csv`),
   плюс `shape` и `columns`. Читать из CSV (не из DataFrame в памяти) —
   это заодно верифицирует корректность записи файла. Без этого шаг считается
   незавершённым. Больше строк — только если пользователь явно попросил.
   Формат — Markdown-таблица для удобства чтения.
8. **Обновление реестра функций:** Claude обновляет файл
   `info/01_functions_implemented.md` (структура — по шаблону
   `manual_forms/02c_FUNCTIONS_IMPLEMENTED.md`):
   - При первичной реализации — добавляет блок о функции (используемый
     API-метод, поля выходного DataFrame с маппингом на оригинальные поля API,
     специфика/сложности, дата в "Истории изменений")
   - При последующих изменениях функции — обновляет соответствующий блок и
     добавляет запись в раздел "История изменений" с датой и причиной
9. После показа результата и обновления реестра — переход к следующей функции

Зачем единый цикл для всех функций: первая функция верифицирует базовую
инфраструктуру (auth, HTTP-обёртки, парсинг envelope, кодировка) — все
последующие опираются на ту же базу. Падать на ней дешевле, чем на пятой.
Spec/plan дают пользователю контроль над каждой функцией без исключений.

## Шаг 5 — ТЗ для внешнего разработчика (опционально)

Когда проект нужно передать разработчику на другом языке (PHP, Go, Java, ...):

1. Открой `tz_templates/HOW_TO_REQUEST_TZ.md` — инструкция по генерации.
2. Скажи Claude: «Сгенерируй ТЗ для {TARGET_LANG}-разработчика по шаблону `test/tz_templates/10_template_tz.md`.»
3. Claude заполнит шаблон и запустит `11_template_generate_tz_pdf.py` → PDF в корне проекта.

**Источники информации для ТЗ — приоритет:** Claude сначала использует
`info/00_api_methods.md` (методы API, поля, формат ответа, лимиты, токены)
и `info/01_functions_implemented.md` (реестр функций: API-метод, маппинг
полей, специфика). Это экономит время и токены — не нужно ходить в интернет
за документацией и искать по всему проекту. К исходникам и интернету —
только если в `info/` данных нет или они устарели.

Шаблоны:
- `tz_templates/10_template_tz.md` — содержательный шаблон ТЗ (Markdown, все разделы с плейсхолдерами)
- `tz_templates/11_template_generate_tz_pdf.py` — скрипт генерации PDF (точка входа для Claude)
- `tz_templates/HOW_TO_REQUEST_TZ.md` — инструкция
