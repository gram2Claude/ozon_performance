# Как сгенерировать ТЗ для внешнего разработчика

## Когда использовать

Когда нужно передать проект разработчику на другом языке (PHP, Go, Java, ...) —
чтобы он мог портировать функциональность без доступа к исходному Python-коду.

---

## Воркфлоу генерации ТЗ

ТЗ — крупный deliverable. Claude **не начинает генерацию без явного утверждения структуры**:

1. Claude предлагает структуру ТЗ (разделы, что войдёт) → пользователь утверждает
2. Claude показывает ключевые решения (функции, примеры, рекомендации) → пользователь утверждает
3. Claude генерирует Markdown + запускает PDF-генератор

---

## Когда готовить ТЗ

ТЗ готовится **в самом конце** — когда все функции реализованы и проверены на реальном API.
К этому моменту у Claude уже есть всё необходимое: реальные примеры запросов/ответов из
истории разработки, заполненные колонки DataFrame, реальные константы тайминга из кода.
Дополнительно ничего готовить не нужно.

## Что нужно подготовить

Перед генерацией убедись, что заполнены:
- `manual_forms/02_NEW_PROJECT_QUESTIONNAIRE.md` — название API, URL, авторизация
- `manual_forms/03_ENTITY_FUNCTIONS.md` — список функций и колонки их DataFrame
- `info/00_api_methods.md` — сводка методов API (заполнена на Шаге 1.5)
- `info/01_functions_implemented.md` — реестр реализованных функций (ведётся на Шаге 4)

Примеры API-ответов, реальные строки таблиц и константы тайминга Claude берёт
самостоятельно из реализованного `{MODULE_NAME}.py` и истории разработки.

---

## Правило приоритета источников при подготовке ТЗ

При сборе материала для ТЗ Claude **в первую очередь** использует файлы из `info/`:

- `info/00_api_methods.md` — endpoints, оригинальные имена и типы полей API,
  формат ответа, rate limits, ограничения диапазона дат, методы аутентификации
- `info/01_functions_implemented.md` — какой API-метод использует каждая
  функция, маппинг полей API → колонки DataFrame, специфика и сложности,
  история изменений

**Why:** эти файлы — заранее подготовленная и провалидированная сводка по
проекту. Использование их избавляет от:
- походов в интернет за документацией API (экономия времени и токенов)
- сплошного поиска по `{MODULE_NAME}.py`, спекам, планам и истории чата

**How to apply:** если нужная информация есть в `info/00_api_methods.md` или
`info/01_functions_implemented.md` — берём её оттуда. К документации в
интернете и к исходникам обращаемся только если в этих файлах данных нет
или они устарели (тогда — обновить файлы перед генерацией ТЗ).

## Команда для генерации

Скопируй промпт, замени `{TARGET_LANG}` (PHP / Go / Java / ...) и отправь Claude:

```
Сгенерируй ТЗ для {TARGET_LANG}-разработчика.

Прочитай (в порядке приоритета — сначала info/, потом остальное):
- info/00_api_methods.md — сводка API: endpoints, поля, типы, формат ответа, лимиты
- info/01_functions_implemented.md — реестр функций: API-метод, маппинг полей, специфика
- {MODULE_NAME}.py — реализация и константы
- CLAUDE.md — архитектура и лимиты API
- manual_forms/02_NEW_PROJECT_QUESTIONNAIRE.md
- manual_forms/03_ENTITY_FUNCTIONS.md
- test/tz_templates/10_template_tz.md — шаблон содержания
- test/tz_templates/11_template_generate_tz_pdf.py — шаблон PDF-генератора

Если нужная информация есть в info/ — бери оттуда. В интернет и в полный поиск
по проекту — только если в info/ данных нет.

Содержание:
- Принципы, типичные ошибки и технические нюансы работы с API,
  которые должен знать разработчик.
- Описание каждой публичной функции, пример ответа API, пример
  таблицы на выходе.
- Сжато, без воды.
- Примеры запросов/ответов и строки таблиц — из готового кода и
  истории разработки. Дополнительно ничего собирать не нужно.

Шаги:
1. Заполни 10_template_tz.md → сохрани как
   specs/TZ_{MODULE_NAME}_{TARGET_LANG}.md
2. По его содержимому заполни 11_template_generate_tz_pdf.py
3. Запусти скрипт → TZ_{MODULE_NAME}_{TARGET_LANG}.pdf в корне проекта
```

При выполнении Claude:
1. Заполнит `{PLACEHOLDER}` в `10_template_tz.md` из анкеты и `03_ENTITY_FUNCTIONS.md`
2. Удалит неподходящие блоки: `[OAUTH ONLY]` / `[API KEY ONLY]` / `[ASYNC ONLY]` / `[SYNC ONLY]` / `[CUMULATIVE]`
3. Подставит реальные примеры API-ответов и таблиц из реализованного `{MODULE_NAME}.py`
4. Запустит `11_template_generate_tz_pdf.py` → получит PDF

## Параметры, которые Claude определяет сам

| Что | Источник (приоритет — `info/`) |
|-----|--------------------------------|
| `{API_NAME}`, `{BASE_URL}` | `02_NEW_PROJECT_QUESTIONNAIRE.md` |
| `{MODULE_NAME}`, список функций | `03_ENTITY_FUNCTIONS.md` |
| Endpoints, поля и типы ответа API, формат (json/csv/zip) | `info/00_api_methods.md` |
| Методы аутентификации (выпуск/обновление токенов) | `info/00_api_methods.md` |
| Лимиты API, rate limits, диапазоны дат | `info/00_api_methods.md` (fallback: `CLAUDE.md`) |
| Какой API-метод использует функция | `info/01_functions_implemented.md` |
| Маппинг полей API → колонки DataFrame | `info/01_functions_implemented.md` |
| Специфика/сложности реализации функции | `info/01_functions_implemented.md` |
| Константы тайминга | константы в `{MODULE_NAME}.py` |
| Примеры запросов/ответов API | из реализованного `{MODULE_NAME}.py` и истории разработки |
| Реальные строки таблиц-примеров | из CSV-выгрузок в `raw_data/` или истории разработки |

---

## Выходной файл

`TZ_{MODULE_NAME}_{TARGET_LANG}.pdf` — в корне проекта.

Также рекомендуется сохранить заполненный `10_template_tz.md` как
`specs/TZ_{MODULE_NAME}_{TARGET_LANG}.md` — для версионирования.
