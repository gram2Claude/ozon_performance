# ТЗ: Ozon Performance API — портирование на PHP

Эталонная реализация: `ozon_performance.py` (Python).  
Цель — портировать функциональность на PHP с сохранением структуры выходных данных.

---

## 1. Цель проекта

Разработать PHP-библиотеку для работы с Ozon Performance API.
Библиотека загружает справочник рекламных кампаний и собирает дневную статистику
(показы, клики, расход, охват, видео-метрики).

Выход каждой публичной функции — структурированный массив ассоциативных массивов,
эквивалентный pandas DataFrame в Python-эталоне.

---

## 2. Общая логика работы библиотеки

Все функции статистики работают по одной схеме:

1. **Получить справочник кампаний.**
   Один запрос `GET /api/client/campaign` возвращает все кампании аккаунта (88 штук).
   Из каждой берём `id` (привести к строке) и `advObjectType` (тип: BANNER / VIDEO_BANNER / ...).

2. **Сформировать список дат.**
   Из параметров `date_from` / `date_to` — массив отдельных дней (YYYY-MM-DD).

3. **Разбить кампании на батчи.**
   API принимает максимум 10 кампаний за раз → делим список на группы по 10.

4. **Для каждой пары (день, батч) — запросить статистику.**
   Отправляем `POST /api/client/statistics` со списком ID кампаний и одним днём.
   API не отдаёт данные сразу — возвращает UUID задачи.

5. **Ждать готовности отчёта (polling).**
   Каждые 7 секунд опрашиваем `GET /api/client/statistics/{UUID}`.
   Когда `state = OK` — переходим к скачиванию.

6. **Скачать и распаковать отчёт.**
   `GET /api/client/statistics/report?UUID=...` → CSV-файл или ZIP с CSV-файлами
   (по одному на кампанию). Распаковываем, привязываем каждый CSV к своей кампании.

7. **Распарсить CSV.**
   CSV содержит строки уровня объявления (одна строка = одно объявление × день).
   В зависимости от функции:
   - `get_campaigns_daily_stat` — суммировать строки по (date, campaign_id)
   - `get_ads_daily_stat` / `get_video_ads_daily_stat` — оставить как есть
   - `get_reach_*` — взять строку «Всего» (охват) или строки объявлений

8. **Собрать итоговый результат.**
   Все строки из всех дней и батчей объединить в один массив с фиксированными колонками.

**Для охват-функций** (`get_reach_campaigns_daily_stat`, `get_reach_ads_daily_stat`)
логика немного отличается: для каждого дня D запрос делается за период
`[global_start_date, D]` (не один день), а из ответа берётся кумулятивный охват.
Дневной прирост (`increment`) вычисляется локально как разница между соседними днями.

## 3. Ключевые принципы работы с Ozon Performance API

### 3.1. Авторизация

- OAuth 2.0, `grant_type=client_credentials`.
  `POST https://api-performance.ozon.ru/api/client/token`
  с `Content-Type: application/json` и телом:
  ```json
  {"client_id": "...", "client_secret": "...", "grant_type": "client_credentials"}
  ```
  **Важно:** `GET /api/client/token` возвращает 405 — только POST.
- Ответ: `{"access_token": "eyJ...", "expires_in": 1800, "token_type": "Bearer"}`.
- Заголовок всех запросов: `Authorization: Bearer <access_token>`.
- Токен обновлять **заранее — за 60 сек до истечения** (`TOKEN_REFRESH_LEEWAY_SEC = 60`),
  иначе протухнет в середине длинного цикла.
- Учётные данные: переменные окружения `CLIENT_ID` / `CLIENT_SECRET`.
  Формат CLIENT_ID: `94252485-1777222314558@advertising.performance.ozon.ru`

### 3.2. Базовый URL и используемые эндпоинты

`BASE_URL = https://api-performance.ozon.ru`

| Метод | Путь | Назначение |
|-------|------|-----------|
| POST | `/api/client/token` | Выпуск access-токена |
| GET | `/api/client/campaign` | Список всех кампаний аккаунта |
| POST | `/api/client/statistics` | Запрос отчёта статистики (возвращает UUID) |
| GET | `/api/client/statistics/{UUID}` | Статус готовности отчёта |
| GET | `/api/client/statistics/report` | Скачивание готового отчёта |

### 3.3. Асинхронная схема получения отчётов (submit → poll → download)

Все методы статистики работают по схеме **submit → poll → download**:

1. `POST /api/client/statistics` → получить `UUID` задачи.
2. Опрашивать `GET /api/client/statistics/{UUID}` каждые **7 сек**
   (макс. **40 попыток** ≈ 5 мин).
   Поле статуса: `state` — значения: `NOT_STARTED` / `IN_PROGRESS` / `OK` / `ERROR`.
3. После `state = OK`: скачать `GET /api/client/statistics/report?UUID=...`
   → ответ `text/csv` (1 кампания) или `application/zip` (2+ кампании).
   ZIP содержит по одному CSV на кампанию.

### 3.4. Лимиты API — критичны, несоблюдение приводит к 429 / блокировке

| Лимит | Значение | Как соблюдать |
|-------|----------|---------------|
| Кампаний в одном запросе | ≤ 10 | Разбивать список на батчи по 10 |
| Параллельных задач генерации | ≤ 3 | Не submit'ить при 3 активных UUID |
| Скачиваний отчётов в сутки | ≤ 1000 | Учитывать при больших периодах |
| Запросов в сутки на аккаунт | ≤ 100 000 | Общий лимит (включая токен) |
| Интервал между submit'ами | ≥ 3 сек | Выдерживать паузу в коде |

### 3.5. Обработка ошибок и rate-limit

- **HTTP 429** — повторить с экспоненциальным backoff:
  стартовая пауза **10 сек**, удвоение, до **5 повторов**.
  Уважать заголовок `Retry-After` если присутствует.
- **Пейсинг submit'ов**: выдерживать ≥ 3 сек между POST `/api/client/statistics` —
  даже при последовательной работе сервер может засчитывать предыдущую задачу активной.
- **Лимит параллельных задач**: если активных UUID уже 3 —
  ждать освобождения слота через polling статуса перед новым submit.
- **`state = ERROR`** в polling — логировать предупреждение, пропустить батч,
  продолжить цикл.
- **Ошибки на уровне батча** (HTTP-ошибка / таймаут polling) —
  не прерывать общий цикл. Логировать и продолжать.

### 3.6. Подводные камни форматов данных

- **Числа** в CSV приходят строками с **запятой** как десятичным разделителем:
  `"25833,00"` → `25833.0`; `"92502,60"` → `92502.6`.
  Функция `parseNum($v)`: `str_replace(',', '.', trim(str_replace("\xc2\xa0", '', $v)))` → `(float)`.
- **Даты внутри CSV** приходят в формате `DD.MM.YYYY` (`"24.04.2026"`).
  В параметрах запросов — формат `YYYY-MM-DD`. Необходим конвертер.
- **Кодировка CSV:** cp1251 — декодировать через `mb_convert_encoding($csvContent, 'UTF-8', 'CP1251')` перед парсингом.
- **Разделитель CSV:** `;` (не `,`).
- **Строка 0 CSV:** метаданные кампании (`;Рекламная кампания № …`) — пропускать.
  Заголовок колонок — первая строка, начинающаяся с `День;`.
- **Строка `Всего`:** итоговая строка — пропускать (кроме функций охвата — см. 2.7).
- **Строка `Корректировка`:** корректировка бюджета без даты — пропускать
  (поле `День` не соответствует формату DD.MM.YYYY).
- **`campaign_id` — хранить как строку.** В JSON-ответе `/api/client/campaign`
  поле `id` приходит как integer. Сразу привести к строке: `(string)$campaign['id']`.
- **Envelope списка кампаний:** ответ `GET /api/client/campaign` приходит как
  `{"list": [...]}`. Если `list` отсутствует — проверить на bare array.

### 3.7. Особый случай — кумулятивная метрика «reach» (охват)

`reach` — количество **уникальных пользователей** за период.
**Нельзя суммировать по дням** — за это отвечает API через `groupBy=NO_GROUP_BY`
и кумулятивный запрос `[global_start_date, D]`.

Чтобы получить значение за день D:
- Отправить запрос с `dateFrom=global_start_date`, `dateTo=D`, `groupBy=NO_GROUP_BY`.
- Для campaign-level (4.4) — взять строку **«Всего»** из CSV → колонка `Охват`
  (API-дедуплицированный охват на уровне кампании).
- Для ad-level (4.5) — взять ad-level строки CSV (одна строка на платформу)
  и просуммировать `Охват` по платформам в рамках одного объявления.
  Это допустимое упрощение — точная дедупликация на уровне (объявление × платформа)
  API не возвращает.

Дневной прирост вычисляется локально:
`increment[D] = reach[D] − reach[D−1]` в рамках одного `campaign_id`.
Для первого дня диапазона: `increment = reach`.

Количество запросов: `N_days × ceil(N_campaigns / 10)`.

## 4. Публичные функции

### 4.1. `get_campaign_dict()`

Справочник рекламных кампаний аккаунта.

HTTP: `GET /api/client/campaign`  
Тело ответа: `{"list": [{id, title, state, advObjectType, fromDate, toDate, ...}]}`.  
Пагинация отсутствует — весь список приходит за один запрос.

| Поле | Тип | Источник в API (ключ) |
|------|-----|-----------------------|
| `campaign_id` | string | `id` (integer в JSON → string) |
| `campaign_name` | string | `title` |

---

### 4.2. `get_campaigns_daily_stat(date_from, date_to)`

Статистика по рекламным кампаниям (без охватов) в разбивке по дням.

Гранулярность: одна строка на `campaign_id × date`.

HTTP: `POST /api/client/statistics` с `groupBy=DATE`.
Запросы: по 1 дню (`dateFrom == dateTo`), по 10 кампаний в батче.

**Агрегация:** CSV содержит ad-level строки (одна строка = одно объявление).
Суммировать `Показы`, `Клики`, `Расход, ₽, с НДС` по группе `(date, campaign_id)`.

| Поле | Тип | Источник (колонка CSV) |
|------|-----|------------------------|
| `date` | string YYYY-MM-DD | `День` (DD.MM.YYYY → конвертировать) |
| `campaign_id` | string | из контекста батча (не из CSV) |
| `views` | float | `Показы` |
| `clicks` | float | `Клики` |
| `money_spent` | float | `Расход, ₽, с НДС` |

---

### 4.3. `get_ads_daily_stat(date_from, date_to)`

Статистика по рекламным объявлениям (без охватов) в разбивке по дням.

Гранулярность: одна строка на `ad_id × date`.

HTTP: тот же `POST /api/client/statistics` с `groupBy=DATE`.
**Агрегация не нужна** — одна строка CSV = одна строка результата.

| Поле | Тип | Источник (колонка CSV) |
|------|-----|------------------------|
| `date` | string YYYY-MM-DD | `День` |
| `campaign_id` | string | из контекста батча |
| `ad_id` | string | `ID баннера` |
| `ad_name` | string | `Название` |
| `views` | float | `Показы` |
| `clicks` | float | `Клики` |
| `money_spent` | float | `Расход, ₽, с НДС` |

Пропускать: строки с пустым `ID баннера`, строки `Всего`, строки `Корректировка`.

---

### 4.4. `get_reach_campaigns_daily_stat(global_start_date, date_from, date_to)`

Охват рекламных кампаний накопительным итогом по дням.

Гранулярность: одна строка на `campaign_id × date`.

Для каждого дня D из `[date_from, date_to]`:
- `POST /api/client/statistics` с `dateFrom=global_start_date`, `dateTo=D`, `groupBy=NO_GROUP_BY`.
- Из CSV взять строку `Всего` → значение колонки `Охват`.

| Поле | Тип | Источник |
|------|-----|----------|
| `date` | string YYYY-MM-DD | параметр D цикла |
| `campaign_id` | string | из контекста батча |
| `reach` | float | `Охват` из строки «Всего» CSV |
| `increment` | float | `reach[D] − reach[D−1]`; для первого дня = `reach` |

Кэш CSV: `reach_{global_start_date}_{campaign_id}_{day}.csv` — не пересекается
с кэшем функций 4.2/4.3 (статистика по дням использует префикс `raw_`).

---

### 4.5. `get_reach_ads_daily_stat(global_start_date, date_from, date_to)`

Охват рекламных объявлений накопительным итогом по дням.

Гранулярность: одна строка на `ad_id × date`. Reach суммируется по всем платформам.

Тот же запрос что 4.4 (`groupBy=NO_GROUP_BY`), тот же кэш `reach_*.csv`.
Отличие: брать ad-level строки CSV (НЕ строку «Всего»), затем суммировать reach по платформам.

| Поле | Тип | Источник (колонка CSV) |
|------|-----|------------------------|
| `date` | string YYYY-MM-DD | параметр D цикла |
| `campaign_id` | string | из контекста батча |
| `ad_id` | string | `ID баннера` |
| `ad_name` | string | `Название` |
| `reach` | float | `Охват`, сумма по платформам |
| `increment` | float | `reach[D] − reach[D−1]` по группе `(campaign_id, ad_id)` |

Пропускать: строку «Всего», строки с пустым `ID баннера`, строку «Корректировка».
Включать только строки с `reach > 0`.

---

### 4.6. `get_video_ads_daily_stat(date_from, date_to)`

Видео-статистика по рекламным объявлениям по дням.
Только кампании с `advObjectType == "VIDEO_BANNER"`.

HTTP: `POST /api/client/statistics` с `groupBy=DATE` (тот же endpoint что 4.2/4.3).
Фильтрация кампаний до VIDEO_BANNER выполняется **до** отправки запросов.

**Отличия от BANNER-кампаний:**
- Колонка расхода: `Расход, ₽` (без «с НДС»)
- Нет колонки `Платформа`
- Есть дополнительные видео-метрики

| Поле | Тип | Источник (колонка CSV) |
|------|-----|------------------------|
| `date` | string YYYY-MM-DD | `День` |
| `campaign_id` | string | из контекста батча |
| `ad_id` | string | `ID баннера` |
| `ad_name` | string | `Название` |
| `views` | float | `Показы` |
| `viewable_views` | float | `Видимые показы` |
| `clicks` | float | `Клики` |
| `quartile_25` | float | `Досмотры по квартилям 25%` |
| `quartile_50` | float | `Досмотры по квартилям 50%` |
| `quartile_75` | float | `Досмотры по квартилям 75%` |
| `quartile_100` | float | `Досмотры по квартилям 100%` |
| `views_with_sound` | float | `Просмотры со звуком` |
| `money_spent` | float | `Расход, ₽` |

## 5. Алгоритм сбора статистики (общий шаблон)

```
1. GET /api/client/campaign → массив кампаний.
   Для get_video_ads_daily_stat: оставить только advObjectType == "VIDEO_BANNER".

2. Сформировать список дат [date_from..date_to] (YYYY-MM-DD).

3. Разбить кампании на батчи по 10 (BATCH_SIZE = 10).

4. Для каждой пары (day, batch):
   a. Проверить кэш: если для каждого campaign_id в батче уже есть
      raw_{date_from}_{date_to}_{campaign_id}_{day}.csv — читать из кэша, пропустить b–f.
   b. Выдержать >= 3 сек с последнего submit.
   c. POST /api/client/statistics:
      { "campaigns": ["id1","id2",...], "dateFrom": day, "dateTo": day, "groupBy": "DATE" }
      → получить UUID.
   d. Polling каждые 7 сек: GET /api/client/statistics/{UUID}
      Проверить поле state:
        NOT_STARTED / IN_PROGRESS → продолжать опрос
        OK → переходить к скачиванию
        ERROR → логировать, пропустить батч
      Макс. 40 попыток (~5 мин) → таймаут, пропустить батч.
   e. GET /api/client/statistics/report?UUID={UUID}
      Content-Type: text/csv → один файл
      Content-Type: application/zip → распаковать ZIP, внутри CSV по одному на кампанию
   f. Кэшировать каждый CSV: raw_{date_from}_{date_to}_{campaign_id}_{day}.csv
   g. Для каждого (csv_bytes, campaign_id): вызвать парсер → список строк.

5. Собрать все строки, привести к фиксированному набору колонок.

Для охват-функций (4.4, 4.5):
   Шаг 4c: groupBy=NO_GROUP_BY, dateFrom=global_start_date, dateTo=day.
   Кэш: reach_{global_start_date}_{campaign_id}_{day}.csv (отдельный namespace).
   Парсер: get_reach_campaigns — строка «Всего»; get_reach_ads — ad-level строки.
   После сбора: вычислить increment через diff по (campaign_id) или (campaign_id, ad_id).
   Первый день: increment = reach (fillna).
```

```

Content-Type: application/json

{
  "client_id":     "94252485-1777222314558@advertising.performance.ozon.ru",
  "client_secret": "<secret>",
  "grant_type":    "client_credentials"
}

Ответ:
{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
  "expires_in":   1800,
  "token_type":   "Bearer"
}
```

```
POST https://api-performance.ozon.ru/api/client/statistics

  "campaigns": ["24251481", "24296538"],

```

## 8. Рекомендации по реализации на PHP

- **Версия**: PHP ≥ 8.1. Composer-проект.
- **HTTP-клиент**: `guzzlehttp/guzzle` с retry-middleware для 429.
  ```php
  $stack->push(Middleware::retry(function ($retries, $req, $res) {
      return $retries < 5 && $res && $res->getStatusCode() === 429;
  }, fn($retries) => 10000 * (2 ** $retries)));  // 10s, 20s, 40s...
  ```
- **Кэширование токена**: приватные поля `$token`, `$expiresAt` в классе.
  Метод `ensureToken()` перед каждым запросом: обновлять если `time() >= $expiresAt - 60`.
- **Парсинг чисел** `parseNum($v)`:
  ```php
  static function parseNum(string $v): float {
      return (float) str_replace(',', '.', str_replace(["\xc2\xa0", ' '], '', trim($v)));
  }
  ```
- **Парсинг дат CSV** `parseDateStr($v)` → YYYY-MM-DD:
  ```php
  static function parseDateStr(string $v): ?string {
      if (preg_match('/^(\d{2})\.(\d{2})\.(\d{4})$/', trim($v), $m))
          return "{$m[3]}-{$m[2]}-{$m[1]}";
      return null;  // "Корректировка" и прочие — возвращает null
  }
  ```
- **Распаковка ZIP**: `ZipArchive` → перебрать файлы, привязать к campaign_id
  по имени файла (обычно `{campaign_id}.csv`).
- **Декодирование cp1251**: `mb_convert_encoding($csvContent, 'UTF-8', 'CP1251')` перед парсингом.
- **Поиск заголовка в CSV**: перебирать строки, найти первую начинающуюся с `День;`.
  Строка 0 — всегда метаданные (`;Рекламная кампания № ...`), пропускать.
- **Прогресс**: вывод в `STDERR` или `symfony/console` ProgressBar.
- **Логирование**: PSR-3 / `monolog/monolog`. Все skip-предупреждения уровня `warning`.
- **Тайминги** — использовать как константы:

  | Константа | Значение |
  |-----------|---------|
  | `POLL_INTERVAL_SEC` | 7 |
  | `POLL_MAX_ATTEMPTS` | 40 |
  | `TOKEN_REFRESH_LEEWAY_SEC` | 60 |
  | `RATE_LIMIT_BASE_SEC` | 10 |
  | `RATE_LIMIT_RETRY_MAX` | 5 |
  | `MIN_SUBMIT_INTERVAL_SEC` | 3 |
  | `MAX_CONCURRENT` | 3 |
  | `BATCH_SIZE` | 10 |

- **Тесты**: моки HTTP-клиента на все 6 публичных функций +
  тест retry на 429 + тест таймаута polling.

## 9. Критерии приёмки

- [ ] Все 6 публичных функций возвращают данные с теми же колонками
      и в том же порядке, что в Python-эталоне `ozon_performance.py`.
- [ ] Корректная работа на реальных учётных данных при 88 кампаниях
      и периоде > 7 дней без ошибок 429.
- [ ] Токен переподписывается автоматически при многочасовом цикле.
- [ ] Лимит 3 параллельных UUID не превышается.
- [ ] Между submit-запросами выдерживается интервал ≥ 3 сек.
- [ ] Числа парсятся корректно (запятая → точка), CSV декодируется из cp1251,
      даты приводятся к ISO YYYY-MM-DD.
- [ ] При сбое любого батча процесс продолжает работать, ошибка попадает в лог.
- [ ] Для первого дня диапазона в функциях охвата `increment == reach`.
- [ ] Строки `Всего` и `Корректировка` не попадают в результат.
- [ ] `campaign_id` и `ad_id` — всегда строки (не числа).
- [ ] README с примерами вызова + `.env.example`.
