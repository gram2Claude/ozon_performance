# Как сгенерировать demo-notebook

## Что должно быть готово перед генерацией

- Заполнена анкета `manual_forms/02_NEW_PROJECT_QUESTIONNAIRE.md` (A1–A6)
- Заполнен файл `manual_forms/03_ENTITY_FUNCTIONS.md` — все функции с типами и колонками
- Создан `.env` с реальными credentials

---

## Формула запроса

```
Сгенерируй demo-notebook по шаблону test/auto_generated/06_template_demo.ipynb.
Сохрани в {MODULE_NAME}/{MODULE_NAME}_demo.ipynb
```

---

## Что Claude обязан сделать

### 1. Прочитать источники данных

| Плейсхолдер | Источник |
|-------------|----------|
| `{API_NAME}` | Анкета A1 |
| `{MODULE_NAME}` | Анкета A2 |
| `{ENV_VAR_1}`, `{ENV_VAR_2}` | Анкета A5 (тип авторизации) — см. ниже |
| Список функций | `03_ENTITY_FUNCTIONS.md` — все Fn1 |
| Тип каждой функции | `03_ENTITY_FUNCTIONS.md` — Fn2 |
| Oneliner каждой функции | `03_ENTITY_FUNCTIONS.md` — Fn4 |

### 2. Определить ENV_VAR по типу авторизации (Анкета A5)

- **OAuth 2.0 (client_credentials):** `ENV_VAR_1 = CLIENT_ID`, `ENV_VAR_2 = CLIENT_SECRET`  
  → две строки assert
- **API key:** `ENV_VAR_1 = API_KEY`  
  → одна строка assert

**ВАЖНО:** имена переменных — без префикса модуля. Используем `CLIENT_ID`, а
не `OZON_CLIENT_ID` / `{MODULE_NAME}_CLIENT_ID`. Каждый проект — отдельный
`.env`, дополнительный префикс избыточен и расходится с шаблоном
`01_PROJECT_STRUCTURE.md`. Если в анкете или в реализации появилось
`{MODULE}_CLIENT_ID` — это баг, переименовать в `CLIENT_ID`.

### 3. Построить разделы notebook

Один раздел на функцию, порядок: **справочники → статистика → охват**.

| Тип функции (Fn2) | Сигнатура | Шаблон раздела |
|-------------------|-----------|----------------|
| Справочник | `fn()` | вызов без дат, `print(f"Всего: {len(result)}")` — не обращаться к конкретным колонкам, они неизвестны заранее |
| Статистика | `fn(DATE_FROM, DATE_TO)` | вызов с датами, `groupby("date")[метрики].sum()` |
| Охват | `fn(GLOBAL_START_DATE, DATE_FROM, DATE_TO)` | вызов с тремя параметрами, топ по `reach` на `DATE_TO` |

Для охватных функций — добавить предупреждение:
```
> **Важно:** `reach` — кумулятивный показатель. Для каждого дня D запрос за [GLOBAL_START_DATE, D].
> Количество запросов = N_дней × N_батчей. GLOBAL_START_DATE должна быть ≤ DATE_FROM.
```

Если охватных функций нет — ячейку `GLOBAL_START_DATE` и предупреждения не добавлять.

### 4. sys.path

Notebook сохраняется в `{MODULE_NAME}/`. Чтобы найти `{MODULE_NAME}.py` на уровень выше:

```python
import sys
sys.path.insert(0, "..")
from {MODULE_NAME}.{MODULE_NAME} import fn1, fn2, ...
```

Если notebook рядом с модулем (в корне репо) — `sys.path.insert(0, ".")`.

### 5. Блок сохранения в CSV

Для каждой функции — одна строка:
- Справочник: `fn_result.to_csv(f"fn_name_{today}.csv", index=False, encoding="cp1251", errors="replace")`
- Статистика/охват: `fn_result.to_csv(f"fn_name_{DATE_FROM}_{DATE_TO}.csv", index=False, encoding="cp1251", errors="replace")`

`errors="replace"` обязателен: без него pandas молча пишет UTF-8 вместо cp1251.

### 6. Обязательная проверка перед сохранением

Перед записью файла убедиться: **ни один плейсхолдер `{...}` не остался** в коде.

Быстрая проверка — grep по `\{[A-Z_]+\}` в готовом тексте. Если находит — заменить.

## Частые ошибки

| Ошибка | Причина | Исправление |
|--------|---------|-------------|
| `AssertionError: {ENV_VAR_1} отсутствует в .env` | Плейсхолдер не заменён | Заменить на реальное имя переменной из A5 |
| `KeyError: 'state'` (или любая колонка) | Ячейка анализа справочника обращается к несуществующей колонке | Не обращаться к конкретным колонкам в ячейке анализа справочника — только `print(len(...))` |
| `ModuleNotFoundError` | Неправильный sys.path | Проверить путь относительно расположения notebook |
| Охват возвращает 1 строку вместо N | `GLOBAL_START_DATE = DATE_FROM` при одном дне | Нормально для тестового периода 1 день |
