# Сводка методов API — Ozon Performance

**Источник:** https://docs.ozon.ru/api/performance/  
**Базовый URL:** `https://api-performance.ozon.ru`  
**Примечание:** до 15.01.2025 использовался хост `performance.ozon.ru` — теперь устарел.

---

## 1. Аутентификация — выпуск токена

### Метод: `POST /api/client/token`

- **Назначение:** выпуск access-токена по client_id / client_secret
- **Тип авторизации:** client_credentials
- **Формат запроса:** `application/json`
- **Параметры запроса:**

  | Поле | Тип | Обязательное | Описание |
  |------|-----|-------------|----------|
  | `client_id` | string | да | Идентификатор клиента (формат: `XYZ@advertising.performance.ozon.ru`) |
  | `client_secret` | string | да | Секретный ключ клиента |
  | `grant_type` | string | да | Всегда `client_credentials` |

- **Формат ответа:** `application/json`
- **Поля ответа:**

  | Поле | Тип | Описание |
  |------|-----|----------|
  | `access_token` | string | JWT-токен для заголовка Authorization |
  | `expires_in` | integer (секунды) | Время жизни токена (1800 сек = 30 мин) |
  | `token_type` | string | Всегда `Bearer` |

- **TTL токена:** 1800 сек (30 минут). Обновлять за 60 сек до истечения.
- **Заголовок запросов:** `Authorization: Bearer <access_token>`
- **Ограничения:** входит в общий лимит 100 000 запросов/сутки

> ✅ **Подтверждено:** HTTP-метод токена — **POST** с JSON-телом. GET возвращает 405.

## 2. Методы получения статистики

### Метод 2.1: `POST /api/client/statistics`

- **Название в доках:** Statistics — формирование отчёта по кампаниям
- **Назначение:** запуск формирования отчёта статистики; возвращает UUID задачи
- **Тип:** асинхронный (submit → poll → download)
- **Группировка / разрезы:** по дням (`groupBy=DATE`) или суммарно (`groupBy=NO_GROUP_BY`)
- **Параметры запроса (тело POST):**

  | Поле | Тип | Обязательное | Описание / допустимые значения |
  |------|-----|-------------|-------------------------------|
  | `campaigns` | array[string] | да | ID кампаний, макс. **10** в одном запросе |
  | `dateFrom` | string (YYYY-MM-DD) | да | Начало периода |
  | `dateTo` | string (YYYY-MM-DD) | да | Конец периода |
  | `groupBy` | string | нет | `DATE` (по дням) / `NO_GROUP_BY` (итог за период) |

- **Формат ответа submit:** `application/json`
- **Поля ответа submit:**

  | Поле | Тип | Описание |
  |------|-----|----------|
  | `UUID` | string | Идентификатор задачи для polling и скачивания |
  | `state` | string | Начальный статус |
  | `createdAt` | string | Время создания |
  | `updatedAt` | string | Время последнего обновления |
  | `request` | object | Echo запроса |

- **Особенности / ограничения:**
  - Максимум **10 кампаний** в одном запросе → батчинг обязателен
  - Максимум **3 одновременные** задачи генерации → контроль `MAX_CONCURRENT`
  - Максимум **1000 скачиваний** отчётов в сутки
  - Общий лимит: **100 000 запросов/сутки**
  - Burst rate limit: недокументирован → выдерживать ≥ 3 сек между submit-запросами

### Метод 2.2: `GET /api/client/statistics/{UUID}`

- **Назначение:** polling статуса готовности отчёта
- **Тип:** синхронный (polling)
- **Параметры:** `{UUID}` в пути
- **Формат ответа:** `application/json`
- **Поле статуса:** предположительно `state`
- **Значения статуса:**

  | Значение | Описание |
  |----------|----------|
  | `NOT_STARTED` | задача в очереди |
  | `IN_PROGRESS` | отчёт формируется |
  | `OK` | отчёт готов к скачиванию |
  | `ERROR` | ошибка формирования |

> ⚠️ **Open question:** точное имя поля статуса (`state` / `status`) — уточнить реальным запросом.

### Метод 2.3: `GET /api/client/statistics/report`

- **Назначение:** скачивание готового отчёта по UUID
- **Параметры запроса:**

  | Параметр | Тип | Описание |
  |----------|-----|----------|
  | `UUID` | string | UUID задачи из submit-ответа |

- **Формат ответа:**
  - 1 кампания → `text/csv` (один CSV-файл)
  - 2+ кампании → `application/zip` (ZIP с CSV-файлами, по одному на кампанию)

- **Кодировка CSV:** UTF-8 BOM (`\xef\xbb\xbf`) ✅
- **Разделитель CSV:** `;` ✅
- **Структура CSV:**
  - Строка 0: метаданные `;Рекламная кампания № {id}, {name}, период {date_from}-{date_to}`
  - Строка 1: заголовки колонок
  - Строки данных: **ad-level** (одна строка на объявление/баннер в день)
  - Строка `Всего`: итог — пропускать
  - Строка `Корректировка`: корректировка бюджета без даты — пропускать
- **Колонки CSV (`groupBy=DATE`, тип кампании BANNER):**

  | № | Колонка | Тип | Описание |
  |---|---------|-----|----------|
  | 1 | `День` | string (DD.MM.YYYY) | Дата |
  | 2 | `ID баннера` | integer | ID объявления/баннера |
  | 3 | `Название` | string | Название объявления |
  | 4 | `Платформа` | string | Платформа показа |
  | 5 | `Показы` | integer | Показы |
  | 6 | `Клики` | integer | Клики |
  | 7 | `CTR (%)` | number (запятая) | CTR |
  | 8 | `Охват` | integer | Охват (уникальные пользователи) |
  | 9 | `Ср. стоимость 1000 показов, ₽` | number (запятая) | CPM |
  | 10 | `Расход, ₽, с НДС` | number (запятая) | Расход с НДС |

  > ⚠️ Колонки могут отличаться для типов кампаний SKU/SEARCH_PROMO — проверить при реализации функций 3–5.

## 3. Справочные методы (не статистика, используются как вспомогательные)

### Метод 3.1: `GET /api/client/campaign`

- **Назначение:** получение списка всех рекламных кампаний аккаунта
- **Параметры запроса:**

  | Параметр | Тип | Обязательное | Описание |
  |----------|-----|-------------|----------|
  | `advObjectType` | string | нет | Фильтр по типу: `SKU`, `BANNER`, `SEARCH_PROMO` и др. |

- **Формат ответа:** `application/json`
- **Структура ответа:** `{ "list": [ {...}, {...} ] }`
- **Поля объекта кампании:**

  | Поле | Тип | Описание |
  |------|-----|----------|
  | `id` | integer | ID кампании — в ответе число, но используется как string в запросах статистики |
  | `title` | string | Название кампании |
  | `state` | string | `CAMPAIGN_STATE_RUNNING` / `CAMPAIGN_STATE_INACTIVE` / ... |
  | `advObjectType` | string | Тип: `SKU` / `BANNER` / `SEARCH_PROMO` / ... |
  | `fromDate` | string | Дата начала кампании |
  | `toDate` | string | Дата окончания кампании |
  | `dailyBudget` | string | Дневной бюджет |

## 4. Сводная таблица методов

| # | Endpoint | Назначение | Формат ответа | Sync/Async | Ключевые поля |
|---|----------|------------|---------------|------------|---------------|
| 1 | `POST /api/client/token` | Выпуск access-токена | JSON | sync | `access_token`, `expires_in` |
